# AI 手绘生成链路

## 目标

明确从运动轨迹数据到手绘风格图片成品的生成流程，先定义能力边界、输入输出和可替换模块，再决定具体模型供应商与实现细节。

## 当前结论

- 初版目标不是做通用文生图产品，而是围绕“运动轨迹可视化”构建受约束、可重复、可追踪的生成流程。
- 生成链路的基本原则：
  - 轨迹必须清晰可辨认，不能被风格化完全吞没。
  - 同一活动在相同参数下应尽量得到稳定风格，而不是完全不可预测。
  - 任务必须异步执行，并能记录中间状态、失败原因和参数快照。
  - 输入、处理、输出要能映射到现有 `activity_streams`、`art_jobs`、`art_results` 数据结构。
- 初版输出建议聚焦静态图片，降低实时交互复杂度，不在第一阶段做可编辑画布或逐步生成预览。
- 生成链路建议拆成五层：
  - 输入准备层：从活动和轨迹数据中提取可生成输入。
  - 轨迹预处理层：清洗 GPS 数据、归一化路径、计算边界框和采样策略。
  - 视觉上下文构造层：决定画布、比例、主题、标题、副标题、地图抽象程度。
  - 风格生成层：调用外部 AI 或图像处理能力生成手绘风格结果。
  - 后处理与存储层：校验结果、生成缩略图、落库存档、回写任务状态。
- 模型策略建议优先采用“可控模板 + AI 风格化”的方向，而不是完全自由创作：
  - 优先保留轨迹结构的稳定性。
  - 把 AI 主要用于风格表达、纹理、纸张感、笔触感和构图润色。
  - 后续再评估是否加入更自由的艺术海报模式。
- 用户可感知的核心结果应该稳定：
  - 轨迹清晰可辨认。
  - 风格具有手绘感。
  - 活动关键信息有机会以标题、副标题或日期信息呈现。
  - 成品适合下载、分享和后续生成缩略图。

### 推荐输入

- 生成任务的主要输入应来自三部分：
  - 活动摘要：
    例如活动名称、运动类型、开始时间、距离、海拔、地点摘要。
  - 标准化轨迹：
    主要来自 `activity_streams.normalized_path_json`、`bbox_json` 和 `point_count`。
  - 生成参数：
    主要来自 `art_jobs.style_preset`、`prompt_snapshot`、`render_options_json`。
- 首版建议最少输入集合：
  - 活动主键和所属用户。
  - 归一化后的二维轨迹路径。
  - 画布比例。
  - 风格模板。
  - 标题与副标题开关。
- 可选输入集合：
  - 原始起终点位置。
  - 海拔变化摘要。
  - 日期、距离、用时等文案元素。
  - 主题色和纸张风格。

### 推荐处理阶段

#### 1. 输入校验

- 校验活动归属是否属于当前用户。
- 校验活动是否为 `is_generatable=true`。
- 校验 `activity_streams` 是否存在，且点数达到最小阈值。
- 校验同一活动是否已有相同参数的进行中任务，避免重复生成。

#### 2. 轨迹预处理

- 对原始轨迹做基础清洗：
  - 去除空点、重复点和明显异常跳点。
  - 按需要做降采样，避免超长轨迹影响生成成本。
  - 统一坐标归一化到固定画布空间。
- 产出建议包括：
  - `normalized_path_json`
  - 轨迹边界框
  - 起点与终点标记
  - 抽稀后的可绘制路径
- 这一层的目标是让后续模型不直接面对混乱 GPS 数据。

#### 3. 视觉上下文构造

- 根据任务参数构建“可渲染描述”：
  - 画布比例：
    如竖版海报、方图、横版封面。
  - 信息层级：
    是否展示标题、日期、距离、地点。
  - 底图抽象程度：
    接近真实地图、弱地图感、纯轨迹构图。
  - 风格模板：
    如钢笔速写、水彩手账、复古地图、极简海报。
- 推荐做法：
  - 先构造一个基础版式或底稿。
  - 再让 AI 对底稿执行风格化，而不是完全从零生成。

#### 4. 风格生成

- 首版推荐保留两种实现路线的兼容性：
  - 路线 A：模板渲染 + 图像风格化
    先绘制轨迹底稿，再通过 AI 或滤镜增加手绘风格。
  - 路线 B：结构约束式图像生成
    把轨迹、布局和文案信息作为强约束输入给图像模型。
- 默认优先路线：
  - 先做路线 A，更容易保证轨迹可读性和结果稳定性。
- 风格生成层最少应返回：
  - 主图文件或二进制结果。
  - 生成耗时。
  - 失败信息。
  - 供应商或执行策略标识，便于排错与后续 AB 测试。

#### 5. 后处理与存储

- 对结果执行：
  - 图片尺寸与格式校验。
  - 必要的压缩和缩略图生成。
  - 元数据回写。
  - 任务状态更新。
- 建议写入：
  - `art_results.image`
  - `art_results.thumbnail`
  - `art_results.width`
  - `art_results.height`
  - `art_results.metadata_json`
  - `art_jobs.finished_at`
  - `art_jobs.status`
- 如结果未通过质量阈值：
  - 标记任务失败。
  - 保留错误信息和参数快照。
  - 允许用户重试或更换风格后重试。

### 推荐风格模式

- 初版建议只做少量模板，避免风格范围过宽：
  - `sketch`
    偏钢笔或铅笔手绘，强调线条和轨迹结构。
  - `watercolor`
    偏纸张纹理和柔和色块。
  - `poster`
    偏版式海报和信息可读性。
- 默认模板建议：
  - `sketch`
    最适合作为 MVP 默认值，因为最容易突出轨迹线条本身。
- 风格参数建议分两层：
  - 用户可见参数：
    风格模板、画布比例、是否显示标题信息。
  - 系统内部参数：
    采样策略、提示词模板、渲染质量等级、供应商模型名。

### 推荐任务状态拆分

- `art_jobs.status` 建议仍保持对外简洁：
  - `pending`
  - `processing`
  - `succeeded`
  - `failed`
  - `canceled`
- 如需记录生成内部阶段，建议放到内部元数据中，而不是扩张外部状态枚举：
  - `input_validated`
  - `path_normalized`
  - `layout_built`
  - `rendering`
  - `post_processed`
- 这样前端状态保持简单，后端排错仍有足够细节。

### 质量判断建议

- 成功结果至少满足：
  - 图片可正常打开。
  - 尺寸符合预期比例。
  - 轨迹主线存在且可识别。
  - 文件大小和格式适合下载与分享。
- 可增加的自动校验：
  - 轨迹区域占比不能过小。
  - 图片不能是纯空白或异常单色。
  - 标题区和轨迹区不能严重重叠。
- 质量不通过时的策略：
  - 首次自动重试一次。
  - 重试仍失败则进入 `failed`。
  - 前端展示“可重试 / 更换风格后重试”。

### 与现有数据模型的映射

- 输入来源：
  - `activities`
    提供活动摘要与展示文案素材。
  - `activity_streams`
    提供可计算的轨迹主输入。
  - `art_jobs`
    提供任务参数、风格模板和状态跟踪。
- 输出去向：
  - `art_results`
    保存图片、缩略图和结果元数据。
  - `art_jobs`
    保存执行过程、错误信息、开始结束时间和 worker 标识。

### 推荐实现边界

- 前端只负责提交生成参数和查看状态，不直接参与图像生成。
- API 层负责创建任务、校验参数和返回任务状态。
- 后台任务层负责真正的轨迹处理、风格生成和文件落地。
- 模型供应商、提示词模板和风格参数应尽量配置化，避免把策略写死在控制器逻辑中。

## 待细化

- 首版是否完全采用“模板渲染 + 风格化处理”，还是从一开始就接入结构约束式图像生成。
- 输入提示词、风格参数、模板定义方式，以及它们是否需要写入数据库。
- 输出分辨率、比例、文件格式和生成时延目标。
- 是否保留原地图地理信息，还是做更强抽象化艺术处理。
- 质量校验是否只做基础校验，还是引入更复杂的自动审查。
- 图片文件是先存 PocketBase file 字段，还是尽早落到对象存储。
- 是否生成多尺寸结果，例如详情页大图和列表缩略图。
