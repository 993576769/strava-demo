# 用户流程

## 目标

梳理用户从首次进入产品，到同步活动、生成手绘轨迹图，再到查看和分享结果的主路径，为后续页面设计、接口设计和任务流转提供基础。

## 当前结论

- 账号与业务主路径分两段：
  - 先完成产品账号登录。
  - 再完成 Strava 连接、活动同步、图片生成和结果查看。
- 初版主流程建议如下：
  1. 用户进入产品首页，理解产品价值与示例效果。
  2. 用户点击登录入口，当前只展示 GitHub 授权登录。
  3. 用户完成 GitHub OAuth，进入产品内已登录状态。
  4. 系统检查用户是否已连接 Strava；未连接则进入连接引导。
  5. 用户连接 Strava，系统创建首次同步任务。
  6. 前端展示“已连接、同步中”状态，并提示等待活动导入完成。
  7. 用户在活动列表中筛选并选择一条带 GPS 轨迹的活动。
  8. 用户查看活动详情、确认可生成状态并选择风格。
  9. 用户发起手绘轨迹图生成任务。
  10. 系统异步处理任务并返回生成状态。
  11. 用户查看生成结果，并进行下载、分享或再次生成。
- 初版重点覆盖“GitHub 登录成功、Strava 连接成功、可同步、可生成、可查看结果”五个关键节点。
- 跑步、骑行等存在清晰轨迹的活动类型，优先纳入首版主流程。
- GitHub 登录失败、无轨迹数据、Strava 授权过期、同步失败、生成失败等情况，需要进入异常流程说明。

### 当前 MVP 用户路径

#### 路径 A：新用户首次进入

1. 查看产品介绍和示例作品。
2. 使用 GitHub 登录产品。
3. 登录成功后进入空状态页。
4. 空状态页引导用户连接 Strava。
5. Strava 首次同步完成后，进入活动列表。

#### 路径 B：已连接用户日常使用

1. 用户再次进入产品并恢复登录态。
2. 系统读取本地活动列表和最近作品。
3. 如检测到可增量同步，则后台更新活动。
4. 用户选择活动并发起新的生成任务。
5. 用户查看生成结果或历史作品。

#### 路径 C：活动到成品

1. 用户打开活动详情。
2. 系统展示基础活动信息、轨迹可用性和最近生成记录。
3. 用户选择风格模板或默认风格。
4. 提交生成任务。
5. 任务处理中时，页面可轮询或订阅状态。
6. 成功后跳转结果页或在原页展示成品入口。

### 后续扩展用户路径

- 后续新增账号密码体系时，主路径不变，只替换第 2 步登录入口：
  - GitHub 登录。
  - 邮箱/账号密码注册。
  - 邮箱/账号密码登录。
  - 找回密码和重置密码。
- 不论哪种登录方式，登录成功后都应进入同一套产品内用户状态、同一套 Strava 连接流程和同一套活动/生成流程。

### 异常与分支流程

- GitHub 登录失败：
  - 提示授权失败或用户取消授权。
  - 允许重新发起登录，不进入业务主流程。
- 后续账号密码登录失败：
  - 明确区分账号不存在、密码错误、账号禁用等场景的展示策略。
  - 找回密码入口应在登录失败场景中可见。
- Strava 未连接：
  - 登录成功后优先展示连接引导，而不是空活动列表。
- Strava 连接已失效：
  - 已登录用户进入产品后，若连接状态为 `reauthorization_required`，优先提示重新授权。
- 首次同步中：
  - 页面展示处理中状态和刷新提示，不要求用户停留在授权回调页等待。
- 无可生成活动：
  - 给出“无轨迹”或“当前活动类型暂不支持”的解释文案。
- 生成失败：
  - 保留失败原因和重试入口，不丢失用户当前活动上下文。

## 待细化

- 首屏是先强调作品示例，还是先强调“连接 Strava 一键生成”。
- 首页到登录页之间是否需要独立的营销落地页。
- 首次进入是否提供 Demo 示例图，降低理解成本。
- GitHub 登录和后续账号密码登录的入口是并列展示，还是在二级面板中展开。
- 活动选择页是列表优先、地图优先，还是作品导向优先。
- 风格选择是模板化还是完全交给 AI 自由生成。
- 分享链路是否包含公开落地页，仅下载图片是否足够。
