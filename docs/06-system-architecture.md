# 系统架构

## 目标

说明前端、PocketBase、后台任务和 AI 能力之间的职责边界，明确哪些逻辑适合同步处理，哪些逻辑必须异步化。

## 当前结论

- 初版系统建议按五层理解：
  - 前端应用层：登录入口、Strava 连接、活动浏览、任务发起、结果查看。
  - 认证与业务 API 层：承接 GitHub 登录、后续账号密码能力、业务接口聚合。
  - 数据与存储层：PocketBase collection、文件字段和业务状态持久化。
  - 后台任务层：Strava 同步、生成任务执行、失败重试、后续密码恢复通知任务。
  - 外部服务层：GitHub OAuth、Strava API 与 AI 图像能力服务。
- 核心原则：
  - 产品账号认证与 Strava 数据授权必须解耦，不把第三方数据源当作登录系统的一部分。
  - 与第三方 API 通信、耗时图像生成，都不应阻塞前端页面。
  - 用户触发生成后，应立即返回任务状态，再由异步流程推进。
  - 前端优先读取本地已同步活动，而不是实时直接打 Strava。
  - 登录方式可以扩展，但用户、会话和业务数据模型保持统一。
- 当前模板可作为演进基础：
  - Vue 前端保留页面、路由、状态管理结构。
  - PocketBase 保留认证和 collection 管理能力，并优先接入 GitHub OAuth。
  - 需要新增后台任务执行机制，承担同步和生成。
  - 后续增加账号密码注册 / 登录 / 找回密码时，应尽量复用同一用户表和会话结构。

### 分层职责建议

#### 1. 前端应用层

- 负责：
  - 登录入口展示。
  - 当前会话恢复与路由守卫。
  - Strava 连接引导。
  - 活动列表、活动详情、生成任务状态和结果页。
- 不负责：
  - 保存 GitHub 或 Strava 的敏感凭证。
  - 直接访问 Strava API 或 AI 服务。
- 页面状态建议拆分：
  - 未登录。
  - 已登录但未连接 Strava。
  - 已连接且同步中。
  - 已连接且可浏览活动。
  - 任务处理中。
  - 结果可查看。

#### 2. 认证与业务 API 层

- 负责：
  - GitHub OAuth 登录入口与回调承接。
  - `GET /auth/session` 这类统一会话接口。
  - 后续账号密码注册、登录、找回密码、重置密码。
  - Strava 连接、断开、同步触发、活动查询、生成任务创建。
- 设计要求：
  - 当前先支持 GitHub，但接口命名不能把认证方式写死为唯一方案。
  - 后续账号密码登录接入后，应输出与 GitHub 登录一致的会话结构。
  - 所有业务接口都基于“产品账号已认证”这一前提。

#### 3. 数据与存储层

- 负责：
  - `users`、`strava_connections`、`activities`、`activity_streams`、`art_jobs`、`art_results` 的持久化。
  - 文件、图片、任务状态和同步元数据存储。
- 默认策略：
  - PocketBase 继续作为主数据库和文件存储入口。
  - 原始第三方快照与内部规范化字段并存。
  - 未来如果图片或轨迹数据体量过大，再逐步拆到对象存储或独立服务。

#### 4. 后台任务层

- 负责：
  - Strava 首次同步与增量同步。
  - 活动详情与 stream 拉取。
  - 生成任务执行和失败重试。
  - 后续密码找回通知或邮件发送。
- 设计原则：
  - 任务必须幂等。
  - 不把重任务放在前端请求链路中执行。
  - 对 Strava 配额和 AI 调用成本做统一调度。

#### 5. 外部服务层

- GitHub：
  - 当前承担产品登录入口。
  - 未来只作为登录方式之一，而不是唯一身份系统。
- Strava：
  - 提供活动数据来源。
  - OAuth 只用于数据授权，不参与产品登录态恢复。
- AI 服务：
  - 接收规范化轨迹和风格参数，返回图片结果或失败信息。

### 关键数据流

#### 登录到业务访问

1. 前端发起 GitHub 登录。
2. 认证与业务 API 层完成 OAuth 交换并建立产品会话。
3. 前端通过统一会话接口恢复用户状态。
4. 后续无论是 GitHub 还是账号密码登录，都走同一会话读取路径。

#### Strava 连接到活动可见

1. 已登录用户发起 Strava 连接。
2. API 层完成 OAuth 回调并写入 `strava_connections`。
3. 后台任务层启动首次同步。
4. 数据层写入 `activities` 和 `activity_streams`。
5. 前端只从本地数据层读取同步结果。

#### 生成任务到结果可见

1. 用户发起生成。
2. API 层创建 `art_jobs`。
3. 后台任务层调用 AI 服务处理。
4. 数据层写入 `art_results` 并更新 `art_jobs` 状态。
5. 前端轮询或订阅结果状态。

### 演进策略

- 当前阶段：
  - GitHub 作为唯一登录方式。
  - Strava 作为唯一活动来源。
  - PocketBase 作为主数据承载层。
- 下一阶段：
  - 新增账号密码注册、登录、找回密码。
  - 保持 `users` 不变，只扩展认证入口和恢复流程。
- 后续阶段：
  - 可能扩展更多运动平台。
  - 可能把重任务执行层独立成专门 worker 服务。

## 待细化

- 认证与业务 API 层是直接用 PocketBase 原生能力组合，还是增加一层自定义 BFF。
- GitHub 登录与后续账号密码登录在前端路由层如何共存。
- 后台任务层放在 PocketBase hooks、独立 worker，还是外部服务中。
- 前端获取任务状态采用轮询、实时订阅还是混合方案。
- 密码找回邮件、通知和链接校验由哪一层负责。
- 图片存储、缓存和 CDN 方案。
- 部署拓扑是单机优先还是容器化多服务优先。
